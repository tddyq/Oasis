# 局内补人

局内补人是一项用于在游戏对局中，动态向局内补充新玩家的后台匹配机制。

<br>

## 补人流程

补人流程包含三个必要阶段，必须按顺序执行：开启补人-申请补人-补人结束。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/i2XKVimage.png)

**开启补人**

- 调用 [`UGCGameSystem.OpenPlayerJoin`](https://developer.gp.qq.com/api/#/searchContent/UGCGameSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%9F%BA%E7%A1%80%E5%8A%9F%E8%83%BD%2FUGCGameSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCGameSystem&autoJump=OpenPlayerJoin) 开启补人功能
- 开启补人是后续申请的必要前置条件，未开启状态下申请无效
- 功能开启后默认持续生效，无需重复开启

**申请补人**

- 调用 [`UGCGameSystem.ApplyPlayerJoinLimitCount`](https://developer.gp.qq.com/api/#/searchContent/UGCGameSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%9F%BA%E7%A1%80%E5%8A%9F%E8%83%BD%2FUGCGameSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCGameSystem&autoJump=ApplyPlayerJoinLimitCount) 函数提交补人申请
- 函数参数为一个表，格式为 {[队伍ID] = 需求人数, ...}，用于指定为哪些队伍补充多少玩家
- 开发者需主动调用此接口申请补人

**补人结束**

- 自然停止：当补入玩家人数达到申请的总人数时，补人完成，自动停止补人
- 手动终止：可调用 [`UGCGameSystem.StopPlayerJoin`](https://developer.gp.qq.com/api/#/searchContent/UGCGameSystem?classDetailShow=true&path=class%2Fdetail%2F%E5%92%8C%E5%B9%B3%E5%85%A8%E5%B1%80%E6%8E%A5%E5%8F%A3%2F%E5%9F%BA%E7%A1%80%E5%8A%9F%E8%83%BD%2FUGCGameSystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCGameSystem&autoJump=StopPlayerJoin) 主动停止补人，此操作将清空所有补人请求（包含未完成的补人请求），可根据玩法实际情况进行手动关闭

<br>

## 补人规则

补人功能遵循一系列规则与条件，生产环境中根据全局玩家的匹配行为而定，并不保证一定能补人成功。

**生效条件**

匹配池中符合条件的人数必须大于或等于申请补的人数，补人才能生效。若处于匹配池中的玩家人数不足，则本次补人申请将无法执行；否则系统将持续尝试补人，直至已补人数达到申请目标或者命中超时规则。

> 在移动端测试时，开发者需关注申请补人人数与参与匹配人数的对应关系，若申请人数与实际匹配人数不符（例如申请补2人，但仅启动1个客户端尝试加入匹配队列），系统会因匹配超时保底机制，将该客户端强制单独分配至新的对局

**超时规则**

- 在每个对局第一次发起补人请求时，后台开始计时，30分钟内如果未能完成补人任务，则补人的匹配池将被清空，本次补人请求被取消
- 如果一个玩法模式连续1小时没有新玩家进入匹配池，该模式的匹配池数据（包括已有的补人请求）将被清空

**补人叠加**

如果多次提交补人申请，补人请求会被合并处理，对于同一队伍，新的合法申请人数会累加到该队伍已申请且未补人成功的人数上，当凑齐申请的总人数时则补人成功。

**补人上限**

- 申请的补人总人数不能超过玩法匹配中设置的最大玩家数量，即补人为符合 ``队伍数量 * 小队玩家数量`` 总人数条件下的补满逻辑
- 如果玩法启用了 [阶段匹配](https://developer.gp.qq.com/wikieditor/#/catalog/20062)，则最大可补充的玩家人数上限以配置中人数最多的阶段作为计算基准，例如阶段匹配配置了1 * 3 ，1 * 2，1 * 1三个阶段，则允许补充的最大人数为 ``3 - 当前服务器人数``
	![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/10Bg7%E4%BC%81%E4%B8%9A%E5%BE%AE%E4%BF%A1%E6%88%AA%E5%9B%BE_17666457143863.png)

## 案例演示


在游戏对局进行期间，当有玩家离开导致队伍人数空缺时，通过监听玩家离开事件，自动为其所在队伍申请补充1名新玩家，并在新玩家加入时通报。


```lua
function UGCGameMode:ReceiveBeginPlay()
    UGCGameSystem.OpenPlayerJoin()
    UGCGenericMessageSystem.ListenGlobalMessage(self, UGCGenericMessageSystem.Messages.UGC.Player.PlayerExit, self, self.OnPlayerExit)
    UGCGameSystem.ApplyPlayerJoinSucceededDelegate:Add(self.OnPlayerJoinSucceeded, self)
end

function UGCGameMode:OnPlayerExit(PlayerKey)
    ugcprint("有玩家退出")
    local TeamID = UGCTeamSystem.GetTeamIDByPlayerKey(PlayerKey)
    UGCGameSystem.ApplyPlayerJoinLimitCount({[TeamID] = 1})
end

function UGCGameMode:OnPlayerJoinSucceeded(UID, RemainingPlayerCountToJoin)
    ugcprint("UGCGameMode:OnPlayerJoinSucceeded")
    UGCGameSystem.StopPlayerJoin()
end
```
