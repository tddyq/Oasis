# 商业化系统

绿洲启元商业化系统是提供给玩法的付费功能解决方案，开发者可以根据实际的玩法需求，定制适合玩法自身的付费商品（商品形式不限），而玩家则能够使用绿洲官方货币购买相关付费商品，且商品购买后将保存在玩家账户（对局结束也将保留）。

> **商业化权限可联系官方人员进行咨询与申请开通**

<br>

## 系统概述

### 货币及商品购买

绿洲官方货币包括：绿洲币和启元币。
- 绿洲币： 玩家充值获得
- 启元币： 玩家完成绿洲任务获得

玩法接入绿洲启元商业化功能后，玩家可以通过两种方式进行付费商品的购买：

**1. 玩法详情页购买**

无需进入玩法就能够直接购买商品的途径，可支持绿洲币、启元币购买付费商品。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/q2PGKimage.png)

**2. 玩法内购买**

开发者在玩法内可以自行设计商城的购买方式，仅支持绿洲币购买付费商品。

---

### 系统交互流程

商业化系统给玩法提供了获取和修改玩家账户内自定义商品数据的能力：
- 商品购买接口和物品使用接口：玩法可以向玩家账户添加指定的商品和扣除指定物品，添加商品会根据商品的价格信息扣除玩家对应数额的绿洲币
- 数据获取接口：玩法可以获取到玩家账户内于该玩法定义的商品数据和物品数据，玩法可以使用这些数据展示和制作相关的功能

由于对玩家账户的数据读取和修改都需要在游戏后台服务器进行，所以玩法需要先注册购买/使用商品的协议，调用绿洲启元商业化系统的接口时，会根据协议指定的购买/使用的商品ID和数量，向后台服务器发起购买/使用请求：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/6xaVWimage.png)

更多商业化相关的API可以查看 [**UGCCommoditySystem类**](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)

## 系统功能接入

### 商业化数据配置

商品数据依赖于编辑器内的两张配置表：<span style="color:red;">``物品表（UGCObject）``</span> 与 <span style="color:red;">``商品表（UGCShop）``</span>

> 注意区分物品和商品概念：直接售卖的是商品，购买商品后获得的是物品

玩法工程默认情况下不存在以上配置表，需要通过编辑器菜单栏下的 ``窗口 -> Create System Default Table -> Create Product Table`` 手动创建，创建后在工程目录下会自动生成Data目录，两张配置表在``Asset/Data/Table``目录中：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/fzMQnimage.png)

### 功能逻辑接入

商业化的功能逻辑主要包含：
- 调用 ``商品购买接口`` 和 ``物品使用接口``分别进行商品的购买获取和物品的使用消耗
- 绑定 ``商品购买结果委托`` 和 ``物品使用结果委托`` 分别处理商品的购买结果逻辑和物品使用结果逻辑

<br>

~~注册商品购买协议和物品使用协议~~（已废弃）
> <font color='#A52A2A'>注册过购买和使用协议的玩法请保留原接入方式，无需更改</font>

<font color='#DDDDDD'>
商品购买和物品使用行为都是在客户端进行处理，而购买和使用通过发送协议来实现，所以需要先在DS内注册协议后，再从客户端调用接口来触发购买商品和使用物品。
购买/使用协议包含商品/物品的ID以及单次购买/使用的数量，根据玩法需要，设计玩法中可能出现的商品类型与单次购买/使用数量组合。

**商品购买协议**


---注册商品购买协议
---生效范围：S
---@param BuyCommodityCMD string  @使用协议
---@param ProductID int  @商品的ID
---@param Count int  @单次购买的数量
function UGCCommoditySystem.RegBuyCMD(BuyCommodityCMD, ProductID, Count)

- BuyCommodityCMD： 待注册的购买协议，调用购买商品接口时，会根据传入的协议发送商品购买请求
- ProductID： 商品的ID，在商品表（UGCShop）中配置
- Count： 协议发送时，单次购买商品的数量


**物品使用协议**

---注册使用物品协议
---生效范围：S
---@param UseCommodityCMD string  @使用协议
---@param CommodityID int  @物品的ID
---@param Count int  @单次消耗的数量
function UGCCommoditySystem.RegUseCMD(UseCommodityCMD, CommodityID, Count)

- UseCommodityCMD： 待注册的使用协议，调用使用物品接口时，会根据传入的协议发送物品使用请求
- CommodityID： 物品的ID，在物品表（UGCObject）中配置
- Count： 协议发送时，单次使用物品的数量

> **注意：**
商品购买协议注册接口传入的ProductID是商品表中的商品ID，而物品使用协议注册接口传入的CommodityID是物品表的物品ID，注意两者参数的区别
</font>

#### 1. 购买商品
开发者可以根据玩法需求自定义商品信息在游戏内的展示形式，例如制作自己的商店界面来展示商品的图标和价格等信息；商品信息内容需要与物品表、商品表的数据配置保持一致。（若不一致，发生实际行为后将以配置为准）

在需要执行商品购买处调用接口 [``UGCCommoditySystem.BuyUGCCommodity2``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)，例如玩家点击了玩法内的购买按钮时，调用接口后系统会弹出购买的二次确认界面，玩家点击“购买”按钮后，客户端会发送购买协议至后台进行购买处理。

商品购买二次确认弹窗界面：

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/bfY2dimage.png)

如果购买流程正常且商品信息校验正确，会提示购买道具成功；而若购买失败，系统会根据失败类型弹出购买失败的提示，例如绿洲币不足。

> 购买成功提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/2Xg4Simage.png)

> 购买失败提示

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/Zwmilimage.png)

若开发者需要对购买的结果进行额外的逻辑处理，例如根据购买结果更新玩家拥有的商品信息显示，则可以绑定购买结果的委托 [``UGCCommoditySystem.BuyUGCCommodityResultDelegate``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)

```lua
--绑定委托
UGCCommoditySystem.BuyUGCCommodityResultDelegate:Add(OnBuyUGCCommodityResult, Tester)

--委托解绑
UGCCommoditySystem.BuyUGCCommodityResultDelegate:Remove(OnBuyUGCCommodityResult, Tester)
```

#### 2. 使用物品
玩家所拥有的商品信息，在游戏内开发者可以根据玩法需求自定义显示形式，比如展示在特定的UI面板内。

在需要执行物品使用处调用接口 [``UGCCommoditySystem.UseUGCCommodity2``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)，例如玩家点击了玩法内的使用按钮时，调用接口时可以决定是否弹出使用物品的二次确认界面：若不弹，则直接发送协议至后台，进行物品校验的结算；若弹出二次确认界面，则玩家点击使用按钮后才会发送协议。

### 绿洲币的显示与充值

**1. 显示并获取绿洲币余额及充值入口**

- 绿洲币余额和充值界面默认不显示，需要显式调用接口 [``UGCCommoditySystem.ShowRechargeEntryUI``](https://developer.gp.qq.com/api/#/searchContent/UGCCommoditySystem?classDetailShow=true&path=class%2Fdetail%2FOthers%2FUGCCommoditySystem.json&isSelect=1&apiEnc=%5B%22%E7%B1%BB%22%5D&apiLabel=UGCCommoditySystem)，接口将会返回充值入口UI的对象实例

- 充值入口UI将显示玩家绿洲币余额，点击入口的 ``+`` 号，将弹出和平的绿洲币充值界面<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/hduoAimage.png)

**2. 调整充值入口UI（可选）**

开发者可以保存调用接口获取到的UI对象实例，若需要隐藏入口或者调整入口位置，可以调用UI相关的接口对入口UI的对象实例进行操作。

```lua
-- 显示并获取界面对象实例
UGCCommoditySystem.ShowRechargeEntryUI():Then(
	function (Result)
		local UI = Result:Get()
		UI:SetVisibility
	end
)
```

**注意事项**

1. PIE调试和外研线测试期间，充值入口的 ``+`` 号不会显示，也无法弹出绿洲币充值界面
2. 发布上传且开通内购权限的工程会显示绿洲币余额，否则不予显示

### 代码示例

通过以上步骤配置商品数据、绑定委托、调用行为接口之后，商业化功能即接入完成，以下是部分代码示例：

**购买绿洲商品代码示例**

```lua
	-- 在某个UI按钮点击后，调用购买函数
	function ShopUI:BuyItem()
			local Result = UGCCommoditySystem.BuyUGCCommodity2(
					9000001,
					UGCGameSystem.GetUGCResourcesFullPath('Asset/Blueprint/S_Icon.S_Icon'),
					"草药用于治疗，是非常好用的道具",
					1
			)
	end
```

绑定购买商品结果委托与回调逻辑处理

```lua
	-- 购买商品后的结果，通过UGCCommoditySystem.BuyUGCCommodityResultDelegate返回，可以在GameState中增加和移除委托监听
	function UGCGameState:ReceiveBeginPlay()
			UGCGameState.SuperClass.ReceiveBeginPlay(self)
			UGCCommoditySystem.BuyUGCCommodityResultDelegate:Add(self.OnBuyUGCCommodityResult, self)
	end

	function UGCGameState:ReceiveEndPlay()
			UGCGameState.SuperClass.ReceiveEndPlay(self)
			UGCCommoditySystem.BuyUGCCommodityResultDelegate:Remove(self.OnBuyUGCCommodityResult, self)
	end

	--可以通过参数得知玩家物品的变化已经是否购买成功
	function UGCGameState:OnBuyUGCCommodityResult(bSucceeded, PlayerKey, CommodityID, Count, UID, ProductID)
			local Result = string.format(
					"(UGCGameState:OnBuyUGCCommodityResult bSucceeded=%s PlayerKey=%s CommodityID=%s Count=%s)", 
					tostring(bSucceeded), 
					tostring(PlayerKey), 
					tostring(CommodityID), 
					tostring(Count),
					tostring(ProductID)
			)
			ugcprint(Result)
```

## 功能测试

### 编辑器PIE调试
- 编辑器环境下可以在PIE时模拟购买商品、使用商品的功能，按正常流程接入即可在PIE时正常测试
- PIE调试时不会进行购买校验，默认购买成功
- PIE调试环境下购买的商品不会永久存档，每次启动调试都会清空

### 手机端测试
- 使用``测试上传``的工程会进行购买校验，该流程下默认购买失败，因此<span style="color:red;">测试上传的方式不适用于购买流程的测试场景</span>
- 编辑器使用``发布上传``，且在管理平台填写测试账号并开启测试后即可测试购买流程，管理平台操作如下：<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/dub3Wimage.png)
- 手机端测试所需的绿洲币，可以通过测试用GM指令增加：<br>
![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/pJ2hjimage.png)
- 手机端测试购买的商品，以及使用的商品会永久存档；如需反复测试，可以通过测试用GM指令清空所有已购买物品

## 商品提审说明
在绿洲启元管理平台上，商业化玩法与非商业化玩法在提审流程上存在细微差别，请参照以下说明执行提审环节的商品信息确认。

### 商业化玩法提审
**1. 绿洲官方货币可购买商品确认**

提审商业化玩法时，首先将经历“确认绿洲官方货币可购买商品”的环节，在此环节可以预览并确认使用“绿洲币”和“启元币”两种官方货币购买的游戏商品及其详细信息；如果确认没问题，勾选“已确认以上商品信息”则可以进入下一步。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/lANykimage.png)

**2. 非官方货币可购买商品确认**

若项目中有使用开发者自己设计的货币体系（非绿洲官方货币）购买的商品，则需要在绿洲启元编辑器中将相关商品购买货币设置类型为<span style="color:red;">**``其他货币``**</span>，在管理平台提审时，确认完官方货币可购买的商品后，将进入“非绿洲官方货币可购买商品”确认环节，查看并确认相关商品信息；如果确认没问题，勾选“已确认以上商品信息”则可以进入下一步。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/OEmkOimage.png)

**3. 商品版本对比**

在确认商品信息时，可以查看本次上传的商品与之前版本的增补、修改或删除情况。

![image.png](https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/3st3Ximage.png)
