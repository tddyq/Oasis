# 性能优化总览

# 性能优化方向与工具使用

## 1. 性能优化：为什么你的FPS游戏需要这么快的“FPS”？
FPS游戏里的“FPS”不只是指“First-Person Shooter”，还代表“Frames Per Second”！帧率对于FPS游戏的体验非常重要。在紧张刺激的游戏过程中，丝滑的帧率是玩家体验的基础保证。

<br>

## 2. 性能指标：CPU、GPU、内存和网络
### 2.1 CPU – 游戏的大脑得跑得快！
不恰当的游戏逻辑处理很容易导致帧率下降，不过只要我们遵守以下规范，在开发阶段就能避免很多性能问题：
1. 避免Tick中的复杂逻辑：使用基于事件的触发机制，可以减少不必要的重复检查。

2. 使用哈希表查询数据：对于经常查询的配表，使用Hash表(Lua中可使用Table)建立索引可以加快查找速度。

3. 异步资源加载：避免在主线程同步加载资源，避免CPU暂停等待，白白浪费性能。

### 2.2 GPU – 游戏的视觉效果要炫酷又流畅！
GPU性能优化的过程是对画面资源分配的过程，过度或不足的资源分配导致性能浪费或画质下降。不过引擎已经为我们准备了以下方法去优化性能：
1. LOD（细节层次距离）设置：LOD用于控制物品在不同距离下使用不同精度的模型渲染，可以有效降提升GPU资源分配效率，将计算资源尽可能多的分配给离玩家近的物体。不过需要注意的是LOD切换过程中模型变化过大可能会被玩家感知，需要合理配置LOD切换的参数或进一步细分LOD。

2. 渲染剔除：渲染剔除用于控制远处较小物体的渲染开关，对于过小的物体不进行渲染。过大的渲染剔除会导致物品移除时被玩家明显感知，国小的渲染剔除又会导致渲染资源浪费。需要结合美术资产的尺寸与表现合理设置。

3. 合批渲染：合批用于将相同物体的合并为一个，一次DrawCall就能完成所有物体的渲染。但是合批也需要根据情况配置，尽可能将离得近且尺寸差不多的物体合批。如果物体离得远或尺寸差异过大，反而会影响渲染剔除策略，影响性能。

### 2.3 内存 – 不要让你的游戏占用太多空间！
内存是游戏正常运行的硬指标，内存问题不仅会导致游戏闪退，甚至可能导致玩家操作系统死机。大部分内存问题可以概括为以下两点：

1. 内存占用过高：这点很好理解，大部分中低端手机内存较少，如果玩法逻辑分配大量内存或使用精度过高的纹理、贴图导致内存占用超出限制，会导致游戏崩溃。开发者在游戏制作过程需要注意对象引用情况，及时释放不再使用的对象，使用合理的纹理、贴图精度。

2. 避免频繁内存分配：这个在日常开发中较少关注，但是这其实是影响性能的关键要素。高频率的内存分配与释放，会导致内存碎片过多，同时也会给引擎的GC过程带来较高的性能压力。在日常开发中对于需要频繁创建的对象（如抛体）考虑创建对象池并复用，降低内存分配导致的GC压力。

### 2.4 网络 – 保持玩家连接畅通无阻！
网络性能是大家比较容易忽视的点，但这块其实对玩家游戏体验的影响非常大。不合理的网络开销不仅会浪费玩家的网络流量，还会导致玩家无法处理战场信息（包括：移动拉扯、命中反馈不及时等），严重情况下会导致玩家掉线。对于网络性能问题，建议开发者处理好以下内容：

1. 不要使用Reliable的广播：这个操作很容易导致网络拥堵。

2. 使用属性同步代替RPC：对于频繁更新的数据，使用属性同步而非远程过程调用（RPC）不仅可以更节省流量，还能更好的支持断线重连的场景。

## 3. 实践指引：使用工具分析与优化性能问题
### 3.1 测试一：日常随时可开的性能测试
在游戏中可以开启“性能调试信息“实时监测“帧率“、“内存占用”、”DrawCalls“和“三角形数量“四个性能数据，分析游戏过程是否有性能卡点。

<img src="https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/nwkJ3image.png" width = "400" alt="性能测试面板" align=center />

<br>

具体测试方式如下：

**步骤一：** 打开性能调试信息显示功能。在”调试面板“的”调试”选项卡中有”性能调试信息“的开关按钮，开启后屏幕左下方可以看到实时性能数据。

**步骤二：** 记录现场，方便复盘。使用屏幕录制功能记录完整的游戏过程，这样你可以在事后回看，找出问题发生的确切时刻。

**步骤三：** 捕捉关键线索，具体包括：

- FPS突变：检查是否有FPS突变：寻找变化时的玩法逻辑，排查玩法逻辑的问题。
- 内存占用异常：检查是否有异常的对象创建或场景加载导致内存超载。
- DrawCall波动：检查画面内容DrawCall的数量，建议控制在200以下。如果过高需要考虑合批或渲染剔除优化。
- 三角形数量检测：检查是否有场景三角形面数过多，建议控制在20万面以下。如果过高需要考虑对模型进行减面或增加LOD层数。

### 3.2 测试二：对性能数据的深度分析
性能Profile数据可以更精细的分析每一帧的性能使用情况，捕捉游戏体验中不易发现的卡点。

<img src="https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/UdrjIimage.png" width = "400" alt="性能数据分析" align=center />

<br>

具体测试方式如下：

**步骤一：** 开启Profile开关。在“调试面板“的“性能”选项卡中可以打开”记录UGCProfile“开关，打开后游戏进程会在回台自动记录性能数据。

**步骤二：** 记录现场。由于Profile数据不能反映具体玩法逻辑的开销，因此需要使用屏幕录制功能记录完整的游戏过程，配合Profile数据进行分析。

**步骤三：** 获取调试数据。关闭”记录UGCProfile“开关，在游戏目录下寻找Profile的导出文件。在PC上该文件在`ShadowTrackerExtra\Saved\Profiling\CSV`文件夹中，在手机上该文件通常在`Android\data\com.tencent.tmgp.projectg\files\UE4Game\ShadowTrackerExtra\Saved\Profiling\CSV`文件夹中，格式为CSV，每一行是一帧的数据。

**步骤四：** 分析调试数据，检查是否存在异常的时间开销。可以结合步骤二的视频数据与日志数据一起分析，判断异常开销的具体原因。可以检查的数据包括：

- Update：所有Actor的Tick时间
- COUNTS/Update：所有Actor的Tick次数
- Create：Spawn Actor的时间开销
- COUNTS/Create：SpawnActor的次数
- Audio：同步加载的时间开销
- COUNTS/Audio：同步加载的次数

### 3.3 测试三：编辑器里的性能快速排查
在我们编辑器的开发者工具中提供了性能自查工具，开启后它会在屏幕下方展示两个进度条：

<img src="https://cgugc-video-test-1258633575.cos.ap-shanghai.myqcloud.com/wiki_picture/GGvOzimage.png" width = "400" alt="编辑器性能问题排查" align=center />

<br>

- SceneMemory：展示物件的内存开销。
- DrawCallNumber：展示物件的DrawCall数量。

我们可以在场景中移动视角，看看哪里的DrawCall或内存开销异常。不过这种方法仅适用于PIE模式下异常物件排查，不能反映出真实场景下的性能表现。
