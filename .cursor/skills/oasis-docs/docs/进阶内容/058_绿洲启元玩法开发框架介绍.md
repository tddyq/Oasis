# 绿洲启元玩法开发框架介绍

在开发过程中，框架设计的重要性不言而喻，如果事先能对玩法开发的功能模块进行合理的规划和区分，善于利用引擎自带的框架设计，把他们放在引擎框架合理的位置中，可以省去大量设计框架和功能的时间，做到事半功倍，代码结构也可以更加清晰利于维护。

下面会介绍引擎中自带的框架设计都由哪些部分组成，以及他们的作用范围。 

<br>

## DS&Client 服务器与客户端

当一局玩法开始后，会生成一个DS（Dedicated Server），DS是所有数据的权威端，一般来说，几乎所有的Actor都是在DS进行创建，然后同步到Client，在Client生成一份复制品，设置为同步的参数，在DS修改后，也会跟着同步到Client对复制品Actor的参数进行同步修改。

所以玩法逻辑也分为DS逻辑和Client逻辑，从防作弊角度来说，只有DS的数据是可以信任的，Client的所有数据都有可能被内存修改篡改，所以大部分的数据都是依靠DS的数据作为执行依据， 玩法的流程逻辑，阶段数据，击杀逻辑，比分数据，胜利判断等都是在DS执行和管理，部分纯粹表现型的功能，比如UI，则是由DS数据同步至Client，Client执行相关UI逻辑。

<br>

## Actor场景中实体的基本类型

如果需要让某个实体对象在场景中生成，那么就必须继承自Actor类型（这里指的是继承链的最底层必须为Actor）。

Actor包含Location，Rotation，Scale等在场景中确定位置的属性。同时也有网络同步的相关职能，开发者可以通过绑定Lua脚本后，重载脚本中的BeginPlay（创建时触发），EndPlay（移除时触发），Tick（每帧触发）等基本事件来完成自己所需要的功能。

后续介绍的GameMode，GameState，PlayerController，PlayerPawn，PlayerState都是继承自Actor，所以具有以上说的Actor全部功能。

### **GameMode和GameState**

- GameMode 玩法模式模块

GameMode主要用于管理玩法流程，<span style="color:red">**实例仅在DS生成**</span>（在Client无法获取到GameMode），单局玩法中只存在一个实例。是可自定义的类型中，在DS最早生成的实例。

一般用于处理一些仅在DS存在和运行的数据和逻辑，玩法流程的编写可以自行在模式编辑器和GameMode中选择一样。

- GameState 数据模块

GameState主要用于管理玩法全局数据，和GameMode一样，在DS仅存在一份实例，但是每个Client都会存在一份DS的GameState的复制品实例，所以GameState是DS全局数据同步至Client的主要桥梁。

通过属性同步可以把DS的全局玩法数据同步至各个Client，如游戏阶段的倒计时，全局分数面板等常驻参数，即时信息则使用广播RPC广播至各个Client，如全局击杀播报，胜利播报等。

### **PlayerController、PlayerPawn和PlayerState**

- PlayerController 玩家控制器模块

PlayerController主要用于管理单个玩家自身控制相关功能。DS中每一位玩家都拥有一个各自的PlayerController实例，Client中则只有该玩家自身的PlayerController复制品实例，玩家的PlayerController在自己的Client无法获取。

PlayerController适合存放与玩家自身相关而其他玩家不需要知道的参数，如击杀数，分数，金币数等，同时玩家自身相关的功能逻辑也可以在此实现，如购买道具，仅自身可以收到的通知播报等。

同时PlayerController也是玩家Client和Server交互的主要途径，所有由Client需要告知Server的消息，都是由PlayerController进行传递，比如想实现一个购买商品的功能，那么应该是由玩家点击购买按钮，由UI通知PlayerController发送一个ServerRPC，向DS申请购买物品，DS里收到申请后再进行扣款，发放物品等操作。

- PlayerPawn（PlayerCharacter）玩家角色模块

PlayerPawn主要用于管理单个玩家所控制角色相关的功能，一般为玩家所操控的人形单位实体。DS中所有存在的角色实体都对应着一份PlayerPawn实例，Client中由于存在网络同步裁剪，只能获取到自身角色一定范围内的PlayerPawn复制品实例（距离自身太远的角色可能会被裁剪导致无法获取）。

PlayerPawn合适存放玩家角色相关参数，如护甲值，攻击力，血量等，角色强相关的逻辑也推荐在这里实现（如使用道具，使用技能，播放动画等）。

- PlayerState 玩家数据模块

PlayerState主要用于管理需要在队伍中同步的相关参数。在DS中，一个PlayerController实例对应一份PlayerState实例，在Client中则有你同队（TeamID相同）的所有玩家的PlayerState实例（Client无法获取到非队友玩家的PlayerState），用途相对来说局限性比较大。

PlayerState一般仅用于存放所有队友都需要知道，但是敌人不需要知道的参数，如队伍面板需要显示的生命值，需要在小地图显示的坐标位置等）。

<br>

## UI系统

UI是玩家和游戏交互最主要的途径，需要注意的一点是<span style="color:red">**UI仅在Client存在**</span>。

在绿洲启元编辑器的用户控件编辑界面中，开发者可以创建UI的基本布局，在绑定脚本进行UI逻辑的控制，通过Construct，Tick等基础事件开发需要的功能。一般UI仅对DS同步至客户端的展示型数据进行处理和展示，开始购买商品和使用技能等逻辑则是调用Controller的ServerRPC进行申请，UI自身不储存重要的数据和执行重要的逻辑。

<br>

网络同步系统在需要联机的玩法开发中至关重要，只有了解并掌握了网络同步系统的相关功能，才能顺利的进行玩法的开发。网络同步系统具体的介绍和使用方式在 [网络同步系统介绍](https://developer.gp.qq.com/wikieditor/#/catalog/203) 中。

## 部分内容补充

### UID

UID即玩家唯一ID，是玩家在和平中唯一标识ID，类似账号，在整个和平中，每个账号的UID都是独一无二的，主要用于一些玩家数据管理用标识，如存档接口就需要传入玩家UID来进行存档和读档。

获取方法：PlayerState.UID，PlayerController.PlayerUID

### PlayerKey

PlayerKey即玩家Key，是玩家在单局玩法中临时生成的一个标识码，在单局玩法中，该局每个玩家的PlayerKey都是唯一的，是玩法内寻找对应玩家的重要参数，大部分单局玩法获取角色实例，获取角色数据相关功能，都需要用到PlayerKey。

获取方法：PlayerState.PlayerKey，PlayerController.PlayerKey，PlayerCharacter.PlayerKey

### TeamID

TeamID即玩家队伍ID，意味着你该局中你属于哪个队伍，该参数初始值会根据玩法上传时的匹配参数决定，后续也可以通过相关接口在游戏中二次分队，获取小队成员，判断敌我阵营，以及部分和平功能，如小地图队友标记，左上角的队友状态栏等功能，都和TeamID有关。

获取方法：PlayerState.TeamID，PlayerController.TeamID，PlayerCharacter.TeamID
